How a task gets routed & orchestrated

One board doc becomes one git strategy. This traces a task from the ▶ Run click through repo resolution, the three-way routing fork, execution, merge, and deploy — the whole path the runner walks for every task.

1

Enqueue — the board is the message bus

SPA ▶ Run → tasks/{id}.runRequested = true

The browser can't reach your laptop, so the task board is the queue. Clicking ▶ Run in the SPA writes run fields onto the task doc; it never calls the runner directly.

SPA ▶ Run→ setTask(id, {runRequested:true, runId, runMode, runLevel, repoUrl?, feature_branch_id?})
2

Claim — one runner, one task, exactly once

onSnapshot(where runRequested == true) · runTransaction · PID lockfile

A live onSnapshot listener wakes the instant a doc flips. The task is claimed inside a Firestore transaction — so a stray second runner can't double-run it — and a PID lockfile enforces one runner per machine (two on the same worktree paths would stomp each other).

// atomic claim — the loser bails
if(!d.runRequested || d.lastRunId === t.runId) return;   // already taken / already ran
tx.update(ref, { runRequested:false, runStatus:"running", lastRunId:t.runId });
3

Resolve the repo — URL → local clone path

normRepo(url) · REPO_MAP[…] · resolveLocalRepo()

The task carries a repoUrl (and maybe a feature_branch_id). The runner normalizes the URL and looks it up in repo-map.json to find a local checkout — that's what makes worktrees fast (no clone). Unmapped repos fall back to a shallow git clone.

const normRepo = url => url.replace(/^https?:\/\//,"").replace(/^www\./,"")
                        .toLowerCase().replace(/\.git$/,"").replace(/\/$/,"");
const resolveLocalRepo = url => REPO_MAP[normRepo(url)] || null;

4

Route — the three-way fork

precedence: feature branch › external repo › self (gameplan-hq)

Exactly one working tree + git strategy is chosen, in strict precedence. This single decision determines where the code lands and how it gets merged.

const useFeature   = !!fb && !!fbLocalRepo;                          // (1)
const useTargetRepo = !useFeature && !!repoUrl
                     && !(localPath === CONFIG.repoDir);              // (2)
// else → (3) gameplan-hq itself
Path 1 · Feature branch

Commit onto the branch

task has feature_branch_id (branch not yet merged) & its repo is mapped

asetupWorktreeAt() — fresh worktree off origin/<branch> (or cut it from the base on first task)
bClaude edits → mergeTaskInto() merges the task branch onto the feature branch & pushes it
cIsolated from master until the Branches tab → Merge fires squashMergeBranch() → master
Path 2 · External repo

Merge to its default branch

repoUrl set & it isn't gameplan-hq — pulse-crm, Angular frontend, merchantsbi-admin

asetupTargetRepo() — local worktree via repo-map, else git clone --depth=1
bBase = origin/main ?? origin/master (auto-detected per repo)
cClaude edits → mergeTaskInto() merges runner branch → the repo's default branch & pushes
Path 3 · Self

Merge to master

no repoUrl, or it resolves to gameplan-hq itself

asetupWorktree() — fresh worktree + branch off origin/master
bClaude edits directly in this checkout
cintegrate() merges the branch → master & pushes
all three converge at → validate → commit → merge → push (stages 7–9)
5

Gate — the task charter dispatcher

if worktree has .claude/task-charter.md + task-management-dispatcher.md

Generic by worktree content: any repo that ships a charter gets a gatekeeper prepended to the prompt. It classifies the task RED YELLOW GREEN, and blocks load-bearing or always-blocked work before any edit. Today the Angular frontend is the only charter-governed repo.

Rejection is not failure. A block emits DISPATCHER BLOCKED: and makes no changes → the task is marked Stuck / Rejected and pushed back to the submitter with the reason. No commit happens.
6

Run mode — how Claude is launched

task.runMode: auto · interactive · detach
🤖 auto

Headless claude -p (stream-json). The default — runs to completion, runner owns git.

💬 interactive

Headless but can ask you: Q&A bridges through the app via ask-user.mjs; Pause / Continue / Abort honored.

⎗ detach

Opens a live tmux session you attach to and drive by hand. Runner won't commit detached work.

Model & effort come from the task's level: quick→haiku · standard→sonnet · deep→opus.

7

Validate — block broken builds before commit

validateAppSyntax() · validateAngularBuild()

Each is a no-op where it doesn't apply, so the same gate runs on every path:

SPA  node --check on the single-file SPA's embedded JS (gameplan-hq, pulse-crm)
NG  npm run build when an angular.json is present (Angular frontend, merchantsbi-admin)

A failed validation throws → the task goes Stuck, nothing is committed.

8

Merge — serialized, retried, self-healing

serializeMerge() · 6 retries · resolveConflictWithClaude()

Only one merge runs at a time (the merge lock), even with parallel task runs. Each attempt re-syncs to the very latest origin, re-merges, and pushes. If the push is rejected because the branch moved under it, it loops. On a real conflict, Claude resolves it and the merge continues — up to 6 attempts before giving up and keeping the branch for manual merge.

fetch origin→ merge --no-ff→ conflict? → Claude resolves→ push HEAD:branch↺ rejected? re-sync & retry
9

Deploy — master push → Vercel

git integration · ignoreCommand: build only master

A push to master is picked up by Vercel's git integration. To stay under the deploy rate limit, vercel.json's ignoreCommand builds only master and skips every other branch — so feature-branch pushes don't burn deploys.

// vercel.json — exit 1 = build, exit 0 = skip
"ignoreCommand": "[ \"$VERCEL_GIT_COMMIT_REF\" != \"master\" ]"
Not every repo deploys via Vercel. merchantsbi-admin merges to its main branch and ships through its own container pipeline — Dockerfile → k8s/ → nginx — so this Vercel/master step doesn't apply to it.
10

Write back — the board reflects the outcome

setStatus() · runStatus · auto-note → SPA onSnapshot re-renders

The runner writes the result straight back onto the task doc. Because the SPA listens on the same collection, your board updates live — no refresh.

Done

Merged & pushed. Result summary + git note posted to the task.

Stuck

Run errored or a merge couldn't complete — branch kept for manual merge.

Rejected

Charter blocked it before any edit — pushed back with the reason to re-scope.
